# rust模块系统

**基于树状层级的作用域控制**

- 区别于TS的文件系统,一个文件就是一个模块,rust更像是使用作用域来定义模块,然后从根开始寻址到指定模块进行调用

## 定义

- `workspace`: 多个package集合(管理)
- `package`: 包含一个cargo.toml,只能有一个library crate和多个binary crate
- `crate`: 编译单元,默认指库 crate
- `module`: crate内代码组织单位

## 层次结构

```bash
workspace
⨽> package
    ⨽>crate
       ⨽>module
```

## 声明模块

1.  使用`mod`关键字
2.  使用`pub mod`将模块公有

- 一个文件可以有多个mod
- 在文件头部的mod,其实就类似导入(声明)文件

## 导入导出

- `lib.rs`: 所有mod统一在`lib.rs`声明并导出
- `pub/pub mod/pub use`: export 公开/公开模块/重命名公开
- `use`: import 导入,本质是创建路径符号的**快捷方式**

我的理解本质和TS的模块不一样,TS的模块本质是文件系统,rust本质是作用域的控制

## 导入导出最佳实践

1. 结构体 (Struct) 和 枚举 (Enum)
   - 惯例：**直接导入**到当前作用域。
   - 因为结构体和枚举通常代表一个“名词”，它们的名字本身就很有辨识度。
   - 做法：`use std::collections::HashMap;`（直接导入 HashMap）
   - 使用：`let map = HashMap::new();`（不需要写 `collections::HashMap`）

2. 函数 (Function)
   - 惯例：**保留一层父级模块名**。
   - 如果直接把函数名（动词）导入进来，别人看代码时很难分清这是本地函数还是外部库函数。
   - 做法：`use std::fs;`（而不是 `use std::fs::read;`）
   - 使用：`fs::read("file.txt")?;`（一眼看出是文件系统的 read，而不是自定义的函数）

3. Trait (接口)
   - 惯例：**直接导入**，或者导入整个 Prelude。
   - 由于 Trait 方法必须 use 才能调用，通常直接导入。
   - 做法：`use std::io::Read;`
   - 使用：`file.read_to_string(&mut s)?;`（导入后，方法就“挂载”上去了）

## 路径规则

- `crate::`: 从当前crate开始,类似项目根目录
- `self::`: 当前模块,类似`./`
- `super::`: 父模块,类似`../`

规则

- 子可以使用父的,但父不可用子的
- `struct` 里的需要pub导出, `enum` 不需要
- `lib.rs`推荐从crate路径引入模块
- `binary`推荐使用项目包名

## 文件嵌套导出和导入

假设crate目录下有

```bash
# 现代模块分层
src/
├── main.rs   <-- 声明 mod a;
├── a.rs      <-- 声明 pub mod b;
└── a/
    └── b.rs  <-- 存放常量的地方 或者 模块

```

```bash
# 旧版习惯
src/
├── main.rs         <-- 声明 mod a;
└── a/
    ├── mod.rs/a.rs <-- 声明 pub mod b;
    └── b.rs        <-- 存放常量的地方 或者 模块
```

**查找文件的规则**
当你写下 `mod a;` 时，编译器查找 a 模块内容的逻辑如下：
`

1. 第一步： 查找同级的 `a.rs`。
2. 第二步： 如果没找到 `a.rs`，查找同级 `a/mod.rs`（这是旧版习惯）。

### 推荐的项目结构

```bash
src/
├── main.rs         # 1. 根模块：只注册顶层业务 (mod network, mod storage)
├── network.rs      # 2. network 文件夹的“大门”
├── network/        # 3. network 的具体子模块全放这里
│   ├── client.rs
│   └── server.rs
├── storage.rs      # 4. storage 文件夹的“大门”
└── storage/
    ├── mysql.rs
    └── redis.rs
```

1. 在 `src/main.rs` 中：
   你只需要写 `mod network;`。它只管 `src/network.rs`。
2. 在 `src/network.rs` 中：
   你写 `pub mod client;` 和 `pub mod server;`。
   重点来了： 此时编译器会去 `src/network/` 目录下找 `client.rs` 和 `server.rs`。

**为什么这样比 TS 好？**

在 TS 中，文件夹只是个物理容器。在 Rust 中，这种结构带来了两个巨大的好处：

- 权限收口： `network/` 目录下的所有细节，都必须经过 `network.rs` 允许（写上 `pub mod`）才能被外部看到。`network.rs` 就像是这个模块的 API 接口定义面。

- 重导出 (Re-export)： 你可以在 `network.rs` 中写 `pub use self::client::Client;`。这样外部调用者只需要 `use crate::network::Client;`，而不需要知道它具体是在 `client.rs` 还是 `server.rs` 里。

**总结**

推荐做法并不是把文件都堆在 src，而是：

- 用一个 `a.rs` 文件去管理一个 `a/` 文件夹。
- `a.rs` 负责声明子模块（目录结构）。
- `a/` 文件夹负责存放具体的代码实现。

## 关于struct和trait

- 要使用`struct`实现了的`trait`方法,需要导入`struct`的同时也导入`trait`
- 即使外部库已经为它的对象写好了 `impl Trait for Struct`，你在使用时也必须同时引入这两者。
- Prelude 模式: 大多数成熟的 Rust 库（如 tokio, itertools, diesel）都会提供一个预导模块 (Prelude)
  ```rust
  // 一次性把这个库最常用的所有 Trait 都引入进来
  use some_library::prelude::*;
  ```

**原因:**

1. 避免方法名冲突（命名空间管理）:
   - 如果你同时用了两个库，它们都给 `String` 增加了一个 `parse()` 方法
   - 如果 Rust 默认自动加载所有 Trait 方法，编译器就不知道你调用的到底是哪个。通过强制 `use`，你明确了你想用哪个库的功能
2. 编译性能:
   - 如果编译器每次处理一个文件都要扫描项目中成百上千个 Trait 来寻找匹配的方法，编译速度会慢得无法接受
